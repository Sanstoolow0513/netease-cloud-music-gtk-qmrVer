# UI 重构设计方案：桌面优先的三区自适应骨架

日期：2026-07-22 · 状态：实施中 · 作者：kimi（与用户确认方向）

## 0. 为什么不是"网页思维"

网页设计的前提是"路由 + 文档流"：页面随导航替换，布局随内容流动，空间靠滚动无限延伸。
桌面播放器的前提是"工具 + 常驻铬框（chrome）"：

- **播放控制是第一公民**：任何页面、任何尺寸下，播放栏都必须可达、可扫视（glanceable）。
- **导航是浅的**：三个顶层视图（发现/榜单/我的）+ 下钻详情，不存在深层路由。
- **窗口是弹性的**：用户会拖窄、拉宽、最大化、全屏；布局必须在所有宽度档下保持可用，而不是只在"设计稿宽度"下好看。
- **密度是美德**：桌面鼠标精度高，相比触屏 web 可以更紧凑，但节奏（spacing rhythm）必须一致，否则显得廉价。

因此本方案不追求"响应式网页式"的任意流体布局，而是采用**分档自适应（breakpoint tiers）**：
每个宽度档都有明确、稳定、经过设计的布局，档间切换干脆利落。

## 1. 三区骨架（保持，修正其自适应）

```
┌──────────────────────────────────────┐
│ HeaderBar: [‹] 导航切换    🔍 👤 ☰   │  ← 导航与全局操作
├──────────────────────────────────────┤
│                                      │
│   内容区（可滚动，宽度档内 clamp）      │  ← 页面内容
│                                      │
├──────────────────────────────────────┤
│ 播放栏：左信息 · 中控制/进度 · 右操作   │  ← 常驻播放控制（播放时出现）
└──────────────────────────────────────┘
```

### 宽度档（AdwBreakpoint）

| 档位 | 宽度 | 导航形态 | 内容网格 | 播放栏 |
|------|------|----------|----------|--------|
| 窄 | < 760px | 底部 ViewSwitcherBar | 2 列卡片；歌曲列表单列 | 简化：隐去音量/心动/歌手行 |
| 标准 | 760–1400px | Header 内 ViewSwitcher | 4–5 列；歌曲列表约 ≥900sp 起双列 | 完整三段式 |
| 宽/全屏 | > 1400px | Header 内 ViewSwitcher | 发现/我的等页面级 clamp 1280 居中；**榜单歌曲列表双列铺满右栏**（不再对列表套 1280 clamp） | 完整三段式，进度条最长化 |

窗口最小宽度从 750 降为 420（窄档真正可用）。

## 2. 已确诊的结构性缺陷与对策

### 2.1 发现页 banner 空白（根因已查明）

- **假象**：banner 图片下载失败。
- **真相 A**：图片下载正常，但 Windows 上 `glib::user_cache_dir()` 指向 `INetCache`（非 `AppData\Local`），排查时误判为"无缓存"。
- **真相 B（渲染层根因）**：`GtkAspectFrame(obey-child=false)` 包着初始无内容的 `GtkPicture`（图片异步加载），在 `AdwClamp(980)` 内尺寸协商坍缩 → banner 区只剩 ~90px 高，图片即使随后加载完成也没有正确的展示空间。
- **对策**：废弃 AspectFrame 比例锁，改为**固定高度档**（标准 300px / 窄 170px，由断点切换）+ `GtkPicture(ContentFit::Cover)` 裁切填充。解除与 1200×465 下载尺寸的硬耦合；高度不再随窗口宽度线性撑高（全屏时 banner 不会变成巨幕）。

### 2.2 播放栏一排到底，窄窗挤压

- **现状**：单排水平盒，封面 50px 写死、标题 12 字符、右侧 6 按钮永不隐藏，唯一弹性是进度条。
- **对策**：三段式重排——左（封面+标题/歌手，可缩）| 中（前进/播放/下一首 + 时间 + 进度条，居中、hexpand）| 右（喜欢/音量/循环/列表）。窄档由断点隐藏次要按钮（心动、音量）与歌手行。时间格式 `0:00 / 4:32`（修 `0.0 / 0.0`）。

### 2.3 顶层导航无自适应

- **现状**：`AdwViewSwitcherBar` 存在但 `reveal` 恒 false、无 id、无断点 → 永不显示；窄窗靠最小宽度硬兜底，header 切换器被截断（"Disc..."）。
- **对策**：窗口级 AdwBreakpoint（<760）：隐藏 header 切换器、reveal 底部切换栏（libadwaita 标准模式，GNOME 应用一致）。

### 2.4 卡片网格空间浪费

- **现状**：FlowBox `homogeneous=false` + 固定 160px 卡片 + 每子项 12px 外边距 → 1160px 窗口只排 4 列且缝隙不均。
- **对策**：`homogeneous=true`，列数按档走（窄 2 / 标准 4–6），间距统一进 spacing token（卡片间 16px、section 间 24px、页面边距 24/12）。

### 2.5 歌曲行列碰撞与超宽留白

- **曾有问题**：专辑列与时长列共用一盒且都 hexpand → 长专辑名把时长顶出列；榜单单列取消限宽后歌名↔时长过远，再限宽 1280 又两侧大留白。
- **现役对策（2026-07-24）**：`SongListView` 宽屏双列交错、窄屏单列；双列紧凑隐藏专辑列。榜单右栏去掉列表外层 1280 clamp，靠双列铺满；播放列表+歌词页 `max-columns=1`。发现/我的等页的页面级 1280 clamp 仍保留。

### 2.6 窗口几何不持久化、无全屏

- **对策**：GSettings 增加 `window-width` / `window-height` / `window-maximized`，关闭时保存、启动恢复；F11 切换全屏（桌面惯例）。全屏只是"很宽的窗口"，断点系统天然覆盖。

### 2.7 榜单页分栏

- **现状**：GtkPaned 位置不持久、无拖拽暗示、侧栏与内容无分隔。
- **对策**：默认 position=240、`wide-handle`、CSS 给侧栏右边界细分隔线。

## 3. 视觉规范（tokens）

- **间距刻度**：4 / 8 / 12 / 16 / 24 / 32。页面边距 24（窄 12）；section 间距 24；卡片间距 16；行内 8。
- **圆角**：卡片封面 12、banner 16、按钮沿用 adwaita。
- **层级**：section 标题 title-3 + 前置小图标（沿用现状）；页面大标题 title-1（详情页）。
- **色彩**：只用 libadwaita 命名色（modern.css 既有约束），强调色仅用于播放中状态与主操作。
- **暗色**：所有新增样式必须同时在明暗主题验证（banner 占位、悬停态）。

## 4. 非目标（明确不做）

- 不做流体卡片尺寸（封面像素动态缩放）：收益小、Rust 改动面大，下轮再议。
- 不改 Action 总线/页面栈架构（已验证可用）。
- 不动 Linux 专属能力（MPRIS/托盘/桌面歌词）。
- 不引入新的依赖。

## 5. 验证方式

`make dev`（或 `build-aux/windows/dev.ps1`：构建 → 同步/打包带 DLL 的便携包 → 启动）→ 截图矩阵：
500×800（窄）、800×800、1160×820（标准）、最大化（1920×1080）、全屏，明/暗两色，
覆盖发现/榜单/详情/播放中四个状态。额外截图/注入工具可放在不入库的 `_windows/dev/`（含前台窗口安全断言）。

## 6. 实施记录（已验证的坑）

- **`AdwBreakpointBin` 不传播子组件尺寸请求**：必须给 bin 自身设 `width-request`/`height-request`，
  否则在盒式布局里被压到 0（libadwaita 会打 "does not have a minimum size" 警告）。banner 高度档
  因此设在 bin 上而非内部容器。
- **多个断点命中时只应用列表中最后一个**（`current-breakpoint` 语义）：重叠档位（如
  player-controls 的 700sp 与 500sp）的窄档 setter 必须自包含，把宽档已隐藏的项再写一遍。
- **`GtkButton` 没有 `icon-size` 属性**（GTK4），模板里写会直接构建失败。
- **Windows 图片缓存目录**：`glib::user_cache_dir()` → `AppData\Local\Microsoft\Windows\INetCache`，
  不是 `AppData\Local`；banner "下载失败"假象源于此。
- **banner 空白根因**：`GtkAspectFrame(obey-child=false)` + 异步加载的 `GtkPicture` 在
  `AdwClamp` 内尺寸协商坍缩；固定高度档后彻底解决，全屏不再被比例锁撑成巨幕。
- gschema 新增 key 后必须同步重编译 dist 的 `gschemas.compiled`（`make dev` / `dev.ps1` 同步路径已处理），
  否则读取新 key 时启动即崩。
- **榜单歌曲列表**：勿再给 `toplist.ui` 右侧套 1280 clamp 或把 `SongListView` 改回单列无限宽；
  现役是组件内双列 + 断点重排（见 `songlist_view.rs` / AGENTS.md）。
