# “我的”页内容预览重构设计

- 日期：2026-07-20
- 状态：已获用户确认
- 范围：重构登录后的“我的”主页内容、加载状态及四类“更多”导航
- 取代范围：本设计取代 `2026-07-19-unified-main-pages-design.md` 中“我的页快捷入口”与“推荐歌单”部分；其余统一样式规范继续沿用

## 目标

将“我的”页从“6 个快捷图标 + 推荐歌单”改为直接可浏览的个人内容首页。用户无需先进入子页面即可查看每日推荐、收藏歌曲、收藏专辑和收藏歌单的部分内容，并可通过每个分区的“更多”进入现有完整列表页。

## 已确认的关键决策

- 删除全部快捷图标，不保留顶部入口卡片。
- 删除现有“推荐歌单”区块。
- 页面只保留四个分区，顺序为：每日推荐、收藏歌曲、收藏专辑、收藏歌单。
- 采用已选择的“丰富展示”方案：
  - 每日推荐预览 8 首，宽窗口双列、每列 4 首。
  - 收藏歌曲预览 8 首，宽窗口双列、每列 4 首。
  - 收藏专辑预览 10 项。
  - 收藏歌单预览 10 项。
- 窄窗口下两列歌曲列表自动叠为单列，不减少预览数量。
- “更多”进入独立完整页，不在当前页内展开，也不使用弹窗。
- 技术路线采用“`MyPage` 专用预览层 + 复用现有完整页”，不连带重构发现页。

## 页面结构

`data/gtk/my-page.ui` 改为四个纵向分区。每个分区沿用发现页的标题模式：

1. 左侧 symbolic 图标。
2. `title-3` 分区标题。
3. 右侧 flat “更多”按钮。
4. 标题下方分隔线。
5. 独立的加载、内容、空状态和失败状态区域。

每日推荐与收藏歌曲各包含两个 `GtkListBox`。外层使用最多两列、最少一列的响应式容器：宽度足够时两个列表并排，每个列表显示连续 4 首；宽度不足时两个列表上下排列，歌曲顺序仍保持 1–8。

收藏专辑与收藏歌单继续使用 `GtkFlowBox` 和现有 `SongListGridItem` 卡片。专辑显示作者，歌单不显示作者；列数随可用宽度自然变化。

## 预览项交互

歌曲预览复用 `SonglistRow` 的数据与操作行为，但增加首页专用的紧凑模式：

- 保留播放指示、歌曲名、歌手、时长和收藏按钮。
- 隐藏专辑文字与“查看专辑”操作，避免双列下横向内容拥挤。
- 不显示从播放列表移除按钮。
- 点击整行立即发送现有 `Action::AddPlay`。

专辑卡片点击后发送 `Action::ToAlbumPage`；歌单卡片点击后发送 `Action::ToSongListPage`。

四个“更多”按钮分别复用现有导航 Action：

- `ToMyPageDailyRec`
- `ToMyPageHeartbeat`
- `ToMyPageAlbums`
- `ToMyPageSonglist`

完整页继续通过 `PageStack` 导航，返回后仍停留在“我的”页。

## 组件边界

### `MyPage`

`src/gui/my_page.rs` 负责：

- 保存 Action sender。
- 初始化四个分区的加载状态。
- 将歌曲数据均分并填充两个预览列表。
- 填充专辑和歌单 `FlowBox`。
- 处理预览项点击、“更多”和重试回调。
- 在加载、内容、空结果、失败四种状态间切换。

预览逻辑保持在 `MyPage` 内，不嵌入 `SearchSongPage` 或 `SearchSongListPage`，从而避免内外滚动容器、分页 offset 和导航状态互相耦合。

### `SonglistRow`

`src/gui/songlist_row.rs` 增加明确的预览模式设置方法，只切换字段和按钮可见性，不改变完整歌曲列表的默认行为。完整页继续保持现状。

### Window 与 Action 总线

`src/window.rs` 作为应用层与 `MyPage` 的桥接层：

- 将歌曲列表转换为对应的收藏状态。
- 把每个分区的数据或状态更新转发给 `MyPage`。
- 不保存首页分页状态。

`src/application.rs` 继续集中处理 API 请求与页面路由。新增按分区加载、成功更新和失败更新 Action；`InitMyPageRecSongList` 被删除。

## 数据流

登录成功后立即通过窗口桥接方法切换到已登录的“我的”页、清空旧预览并显示四个加载状态。用户收藏歌曲 ID 初始化完成后，无论该请求成功或失败，都触发 `Action::InitMyPage` 开始网络加载；成功时可保证每日推荐的红心状态使用最新收藏集合，失败时仍允许四个分区加载。

`Action::InitMyPage` 派发四个独立加载任务，网络请求在 GLib MainContext 中分别执行：

1. 每日推荐调用 `recommend_songs()`，成功后截取前 8 首。
2. 收藏歌曲调用 `user_song_list(uid, 0, 1)` 获取第一个“喜欢的音乐”歌单，再调用 `song_list_detail()`，成功后截取前 8 首。
3. 收藏专辑调用 `album_sublist(0, 10)`。
4. 收藏歌单调用 `user_song_list(uid, 0, 11)`，安全跳过第一个“喜欢的音乐”歌单，再截取 10 项。

每个任务单独把结果回投给 `MyPage`。一项请求失败不会阻塞或清空其他分区。

点击“更多”后继续由现有完整页重新请求最新数据，不复用首页的截断结果：

- 每日推荐和收藏歌曲获取完整歌曲列表。
- 收藏专辑沿用首批 50 项与现有滚动加载。
- 收藏歌单沿用现有完整请求，但把直接的 `sls[1..]` 改为安全跳过首项，空结果不再触发越界。

## 状态与异常处理

每个分区使用独立状态容器：

- 加载中：显示 spinner，“更多”不可用。
- 成功且非空：显示内容并启用“更多”。
- 成功但为空：显示“暂无内容”，“更多”不可用。
- 请求失败：显示“加载失败”和重试按钮，“更多”不可用，同时发送现有错误 Toast。

重试只重新请求失败分区。删除当前 `InitMyPage` 失败后递归发送自身的无限自动重试行为。

退出登录仍由现有 `my_stack` 切回未登录状态。迟到的异步结果只能更新隐藏的登录页内容，不改变登录状态或主动导航。

## 国际化与样式

- 新增的标题、空状态、失败状态、重试提示和 tooltip 均标记为 gettext 可翻译字符串。
- 继续使用 `modern.css` 的页面间距、分区标题、boxed-list 和卡片规范。
- 如需补充歌曲双列或状态容器样式，只添加“我的”页作用域类，不修改其他页面默认样式。
- 不新增硬编码主题颜色。

## 验证计划

### 静态与构建验证

1. 运行 `cargo fmt --check`。
2. 完成 Cargo 或 Meson 全量构建，确认 Rust、CompositeTemplate 与 gresource 均可加载。
3. 运行现有 Meson 数据校验测试，确认 desktop、metainfo 和 gschema 不受影响。
4. 检查改动文件的 IDE linter 诊断。

### 手工验证

1. 登录后四个分区独立进入加载态并逐项完成。
2. 数量分别不超过 8、8、10、10；不足时按实际数量展示。
3. 宽窗口歌曲为双列，窄窗口自动叠为单列且顺序正确。
4. 歌曲点击播放、收藏按钮更新；专辑和歌单卡片进入正确详情。
5. 四个“更多”进入正确完整页，返回行为正常。
6. 收藏专辑完整页仍可滚动加载；收藏歌单为空或仅有“喜欢的音乐”时不崩溃。
7. 单个接口失败只影响自身分区，重试只刷新该分区。
8. 空结果显示对应空状态并禁用“更多”。
9. 注销后显示未登录页，迟到请求不会重新切回登录页。
10. 明暗主题下标题、卡片、列表和状态提示均可读，无新增 GTK critical 日志。
